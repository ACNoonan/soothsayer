# v1.5 deployment spec — pooled-tail + state-augmentation + BMA-CAViaR per-symbol hybrid

> **SUPERSEDED by v2 / M5 (Mondrian split-conformal by regime), deployed via `M5_REFACTOR.md`.** The v1.5 BMA-CAViaR proposal in this document closed the v1 finite-sample tail ceiling at τ = 0.99 by adding per-symbol CAViaR forecasters and Bayesian model averaging. The v2 / M5 deployment closes the same gap by a different, simpler mechanism: a per-regime conformal quantile pooled across symbols (12 trained scalars), with an OOS-fit multiplicative bump and walk-forward shift (4 + 4 OOS scalars). On the same 1,730-row held-out OOS panel, v2 / M5 hits the τ = 0.99 nominal target with realised $0.990$ and Kupiec $p = 0.942$ (cf. v1.5 BMA's realised $0.986$ at $p = 0.081$), at a tighter parameter budget and substantially lower production-port effort. The v2 / M5 deployment is documented in `reports/methodology_history.md` (2026-05-02 entry), `reports/paper1_coverage_inversion/` §4 + §7.7, and `src/soothsayer/oracle.py` + `crates/soothsayer-oracle/src/{config,oracle}.rs`. This v1.5 spec is retained as historical record of an explored-and-deferred path.

## 1. What v1.5 changes vs current production (v0/v1)

Current production methodology (`src/soothsayer/oracle.py`, see §0 of `reports/methodology_history.md`):

- F1 σ regression: `log|residual| = α + β · log(VIX) + γ_e · earnings + γ_l · long_weekend`
- Calibration surface: per-(symbol, regime) empirical-quantile of standardized residuals
- Buffer schedule: `{0.68: 0.045, 0.85: 0.045, 0.95: 0.020, 0.99: 0.010}`

v1.5 adds three layered upgrades:

1. **Trial 6 (state augmentation)**: F1 σ regression gains a `log(fri_vol_20d)` regressor — `log|residual| = α + β · log(VIX) + γ_v · log(fri_vol_20d) + γ_e · earnings + γ_l · long_weekend`. Default ON.

2. **Trial 3 (pooled-tail)**: at τ ≥ 0.95 the calibration surface is computed by pooling standardized residuals across symbols within (regime, time-window) instead of per-(symbol, regime). New `pool_tail_above_q: float = 0.95` parameter on the surface constructor.

3. **Family A BMA-CAViaR per-symbol hybrid (τ=0.99 only)**: at τ = 0.99 the bound is a Bayesian-model-averaged combination of three forecasters per symbol — the empirical-quantile baseline (1+2 above), CAViaR-SAV, CAViaR-AS — with weights derived from 3-fold cross-validated DQ p-values on the pre-2023 calibration set.

Combined v1.5 scorecard at τ = 0.99 on the 1,730-row held-out OOS panel (`exp_caviar_cv_select.py`):

| metric | v0/v1 production | v1.5 BMA |
|---|---|---|
| Per-symbol DQ rejects | 5 / 10 | **2 / 10** |
| Realised | 0.977 | **0.986** |
| Christoffersen $p_\text{ind}$ | 0.956 | **0.392** |
| Kupiec $p_{uc}$ | 0.000 (rejects) | **0.081** (passes) |
| Mean half-width | 580.8 bps | 510 bps |

All four scorecard thresholds pass for v1.5.

## 2. Code changes by file

### 2.1 `src/soothsayer/backtest/forecasters.py`

**Add CAViaR forecaster module.** New functions:

```python
def caviar_sav_quantiles(params, residuals, q_init): ...
def caviar_as_quantiles(params, residuals, q_init): ...
def fit_caviar_sav(residuals, alpha) -> tuple: ...
def fit_caviar_as(residuals, alpha) -> tuple: ...
```

Source: `scripts/exp_caviar_cv_select.py:caviar_sav_quantiles / caviar_as_quantiles / fit_sav / fit_as`. Same signatures, modulo renaming.

**Extend `empirical_quantiles_f1_loglog` to support state augmentation.** Add a `log_vol_col: str | None = "fri_vol_20d"` parameter that, when set, derives a `log(<vol_col>)` regressor and appends it to the σ regression's design matrix. Default ON (Trial 6 finding).

### 2.2 `src/soothsayer/backtest/calibration.py`

**Add `pool_tail_above_q` to surface construction.** Signature change:

```python
def compute_calibration_surface(
    bounds: pd.DataFrame,
    pool_tail_above_q: float | None = 0.95,
) -> pd.DataFrame: ...
```

When `pool_tail_above_q` is set, calibration-surface rows for `claimed_q >= pool_tail_above_q` are computed by pooling standardized residuals across symbols within (regime, time-window) rather than per-(symbol, regime). Default value 0.95 ships Trial 3.

**Add a CV-selection module.** New file `src/soothsayer/backtest/cv_select.py`:

```python
def fit_caviar_per_symbol_with_cv(
    panel: pd.DataFrame,
    fold_boundaries: list[tuple[date, date, date]],
    target_tau: float = 0.99,
) -> dict[str, dict[str, tuple]]: ...

def compute_bma_weights(
    cv_results: pd.DataFrame,
    weight_floor: float = 1e-3,
) -> dict[str, dict[str, float]]: ...

def caviar_q_sequence_for_symbol(
    residuals: np.ndarray,
    sav_params: tuple, as_params: tuple,
    q_init_lo: float, q_init_hi: float,
) -> dict: ...
```

Source: lift the relevant functions from `scripts/exp_caviar_cv_select.py`.

### 2.3 `src/soothsayer/oracle.py`

**Add the BMA-CAViaR path at τ = 0.99.** New constructor parameter `bma_weights: dict[str, dict[str, float]] | None`. When set, the `Oracle.fair_value()` path at target_τ where the inverted claimed_q ≥ 0.99 computes:

```python
lo = sum(w[m] * lo_m for m in ["baseline", "sav", "as"])
hi = sum(w[m] * hi_m for m in ["baseline", "sav", "as"])
```

where `lo_m / hi_m` are the per-method bounds (baseline from the calibration surface, CAViaR from the per-symbol fitted parameters), and `w` is the BMA weight vector for this symbol.

If `bma_weights` is None, the Oracle behaves identically to v1 (backwards-compatible).

**Receipt schema:** the served `PricePoint` gains a `methodology_stack: list[str]` field listing the components that contributed to the bound (e.g., `["pooled_tail", "state_aug", "bma_caviar:baseline=0.33,sav=0.33,as=0.33"]` for AAPL). This is the calibration receipt — consumers can verify the weights.

### 2.4 `crates/soothsayer-oracle/src/{config,oracle}.rs`

Mirror the Python changes — port the BMA + CAViaR logic to Rust, byte-for-byte. The 75/75 parity test gets new cases for the v1.5 paths.

CAViaR fitting itself runs only at calibration time (offline, Python); the Rust runtime only needs to *evaluate* the fitted CAViaR parameters via the recursive q_t sequence. The optimization stays in Python. This keeps the Rust runtime simple — Rust just needs the BMA weight dict and the CAViaR parameter dict, both serialized into the calibration artefact.

## 3. New artefacts

- `data/processed/v1b_caviar_params.parquet` — per-symbol fitted CAViaR-SAV and CAViaR-AS parameters from the full pre-2023 calibration set.
- `data/processed/v1b_bma_weights.parquet` — per-symbol BMA weights derived from CV.
- `data/processed/v1b_caviar_cv_folds.parquet` — per-fold per-symbol per-method DQ p-values from the CV selection process. Audit trail.

These artefacts are the calibration-time outputs of the new `cv_select.py` module. They get persisted alongside the existing `v1b_bounds.parquet` and consumed by `oracle.py` at serve time.

## 7. What v1.5 explicitly does NOT do

- *Replace the calibration surface architecture* — Trial 5 (CEVT) and Trial 2 (full Mondrian conformal) are v2 territory.
- *Address the AAPL last-mile gap (BMA 2/10 vs oracle 1/10)* — explicit AAPL-specific handling (e.g., earnings-cycle regressor) deferred to v2.
- *Add new data inputs* — CEX stock-perp coverage (scryer wishlist item 45) lands in v2 if HOOD's small-sample diagnosis becomes revisitable with continuous weekend perp data.
- *Change the F0 fallback* — F0_stale (Gaussian, asymmetric variants) untouched.
- *Change the regime labeler* — sub-regime granularity (post-shock, vol-spike-decay) deferred to v2.

## 8. Risk + mitigations

- **CAViaR fits drift over time.** Mitigation: re-fit CAViaR per-symbol on a rolling annual schedule; re-derive BMA weights at each refit. Calibration-cycle entry in the methodology log.
- **BMA weights become stale as new data arrives.** Mitigation: add a `weights_age_days` to the receipt; consumers can flag bands served with weights older than e.g. 180 days.
- **HOOD remains rejecting.** This is a known limitation of v1.5; the disclosure is in §6.4.1 with the §10.2 filter as the next-tested hypothesis.
- **AAPL's BMA equal-weight averaging is sensitive to CV noise.** If a future re-run gives different CV-median p-values for AAPL (different weights), the per-symbol DQ at τ = 0.99 may drift. Mitigation: monitor per-symbol DQ on a rolling basis; alert if AAPL's DQ p-value drops below 0.05 in the deployment window for >2 consecutive months.
